# Economic Indicators Tracker

A live data pipeline tracking 6 US macroeconomic indicators from the FRED API, transformed with dbt, and served through a publicly deployed Streamlit dashboard.

**Live dashboard:** [Economic Indicators Tracker](https://econ-indicators-tracker.streamlit.app/)

---

## Architecture

FRED API -> Python -> Snowflake (RAW) -> dbt -> Snowflake (MART) -> Streamlit

## Pipeline Lineage

Full data lineage from source to dashboard, generated by dbt docs.

[dbt Lineage DAG](docs/lineage_dag.png)


## Indicators Tracked

| Indicator | FRED Series ID | Frequency | Unit |
|---|---|---|---|
| GDP | GDP | Quarterly | Billions USD |
| CPI | CPIAUCSL | Monthly | Index (1982-84 = 100) |
| Unemployment Rate | UNRATE | Monthly | Percent |
| Federal Funds Rate | FEDFUNDS | Monthly | Percent |
| Housing Starts | HOUST | Monthly | Thousands of Units |
| Retail Sales | RSXFS | Monthly | Millions USD |

## dbt Models

### Staging - `stg_fred_observations`
Cleans and standardizes raw FRED data. Casts types, filters null dates. Materialized as a view.

### Mart - `mart_economic_indicators`
Builds on staging to calculate period-on-period change per indicator. For percentage series (UNRATE, FEDFUNDS) this is a percentage point change. For raw value series (GDP, CPIAUCSL, HOUST, RSXFS) this is a true percentage change. Materialized as a table.

## Data Quality

Two layers of data quality checks run automatically on every `dbt build`:

**Structural tests** - not_null and accepted_values on staging and mart models ensure data arrives in the expected shape.

**Business logic tests** - custom singular tests enforce domain rules:
- Unemployment Rate is always between 0% and 25%
- Federal Funds Rate is always between 0% and 30%
- Month-on-month change never exceeds 50% for raw value indicators

**Source freshness** - dbt monitors the raw table load timestamp and warns if data is older than 45 days, errors if older than 60 days. Run with `dbt source freshness`.

## Setup

### Prerequisites
- Python 3.9+
- Snowflake account
- FRED API key - free at https://fred.stlouisfed.org/docs/api/api_key.html

### 1. Clone the repo

    git clone https://github.com/DataChaser/data-engineering-portfolio.git
    cd data-engineering-portfolio/econ-indicators-tracker

### 2. Create a virtual environment and install dependencies

    python -m venv venv
    venv\Scripts\activate
    pip install -r requirements.txt

### 3. Set up credentials

Copy `.env.example` to `.env` and fill in your Snowflake credentials and FRED API key.

### 4. Set up Snowflake

    CREATE DATABASE ECONOMIC_INDICATORS;
    CREATE SCHEMA ECONOMIC_INDICATORS.RAW;
    CREATE SCHEMA ECONOMIC_INDICATORS.STAGING;
    CREATE SCHEMA ECONOMIC_INDICATORS.MART;

### 5. Configure dbt

Add your Snowflake connection to `~/.dbt/profiles.yml`:

    economic_indicators:
      target: dev
      outputs:
        dev:
          type: snowflake
          account: "your_account"
          user: "your_user"
          password: "your_password"
          role: "your_role"
          warehouse: your_warehouse
          database: ECONOMIC_INDICATORS
          schema: STAGING
          threads: 4

### 6. Run the ingestion pipeline

    cd ingestion
    python extract.py
    python load.py

### 7. Run dbt

    cd dbt_project/economic_indicators
    dbt build

### 8. Run the dashboard locally

    cd streamlit_app
    streamlit run app.py

## Key Design Decisions

**Why long format over wide format?**
One row per indicator per date means adding new indicators requires no schema changes - just more rows.

**Why full refresh over incremental loading?**
FRED revises historical data. Truncate-reload on every run ensures the table always reflects the most accurate version of every data point. At ~1700 rows this is trivially fast.

**Why separate staging and mart layers?**
Staging cleans and standardizes once. Mart applies business logic on top. Any future mart models build on staging, not raw.

**Why percentage point change for UNRATE and FEDFUNDS?**
These indicators are already expressed as percentages. A change from 4.0% to 4.3% is +0.3 percentage points - not +7.5%. Percentage point change is how economists and analysts actually talk about these indicators.

**Why materialize mart as a table?**
The dashboard queries it on every load. Materializing as a table means pre-computed results - Streamlit reads stored data rather than recalculating on every visit.
